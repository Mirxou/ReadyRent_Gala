## 4. النظام القضائي (AI Jury / Disputes)

أكثر تطبيق تعقيداً — **16 سيرفس، 12 موديل، 30+ فيو، محرك كامل**:

### دورة النزاع
1. **رفع النزاع** → `initiate_dispute()` مع فحص عاطفي
2. **التوجيه** → `TribunalEngine` يحدد المسار (تحكيم/وساطة)
3. **الوساطة** → جلسات وساطة، عروض تسوية
4. **الإثباتات** → Evidence Vault (WORM - Write Once Read Many)
5. **الحكم** → `AdjudicationService.finalize_judgment()` (مندوب/خبير/تقسيم)
6. **الاستئناف** → Appeal مع مراجعة
7. **السجل العام** → أحكام منقحة بهوية مخفية
8. **المحكمة العليا** → Sovereign Override (Phase 42)

### الميزات الفريدة
- **RiskEngine**: تقييم مخاطر النزاع بناءً على الثقة والسوابق
- **Precedent Search**: محرك تشابه للسوابق القضائية
- **Vault Audit**: تدقيق سلسلة التشفير (hash chain)
- **Public Ledger**: سجل عام شفاف للأحكام
- **Consistency Engine**: فحص اتساق الأحكام مع السوابق

الثغرات الحرجة (Critical)
CRITICAL-01:
مفاتيح تشفير PII مكشوفة في مستودع عام
الملفات backend/.env.pii_keys و backend/.env.new_keys تحتوي على مفاتيح تشفير حقيقية (PII_HASH_KEY, PII_ENCRYPTION_KEY_V1, IP_HASH_KEY) وهي مُرتكبة في المستودع العام منذ أول commit. هذا يعني أن أي شخص قام بزيارة المستودع يمكنه الوصول إلى المفاتيح المستخدمة لتشفير بيانات التعريف الشخصية للمستخدمين. السيناريو الأسوأ: إذا تم تخزين أي بيانات مشفرة باستخدام هذه المفاتيح، فهي الآن غير آمنة ويجب إعادة تشفيرها بمفاتيح جديدة بعد تدوير المفاتيح الحالية.
CRITICAL-02:
حماية CSRF غير فعّالة في API
اختبار test_csrf_token_required_for_post يفشل بشكل متكرر: الخادم يُرجع حالة HTTP 200 عند إرسال طلب POST بدون CSRF token، بينما المطلوب إرجاع 403. هذا يعني أن أي موقع ويب خارجي يمكنه إرسال طلبات POST نيابة عن مستخدم مسجل الدخول دون علمه. في DRF، يُعزى هذا عادةً إلى إعداد REST_FRAMEWORK['DEFAULT_AUTHENTICATION_CLASSES'] الذي يتضمن SessionAuthentication بدون تفعيل CSRFExempt بشكل صحيح أو العكس.
CRITICAL-03:
المستودع عام مع بيانات حساسة
المستودع على github.com/Mirxou/ReadyRent_Gala عام (Public)، ويحتوي بالإضافة إلى مفاتيح PII على ملفات إعدادات قد تحتوي على معلومات البنية التحتية. من المفترض أن يكون أي مشروع بهذا الحجم والتعقيد في مستودع خاص (Private)، مع إزالة أي بيانات حساسة من تاريخ Git بالكامل باستخدام git filter-repo.

🔴 1. SÉCURITÉ — Failles critiques confirmées
1.1 Clés cryptographiques commitées en clair, depuis le premier commit
Copybackend/.env.new_keys   → PII_HASH_KEY, IP_HASH_KEY, PII_ENCRYPTION_KEY_V1
backend/.env.pii_keys   → mêmes clés (valeurs différentes, doublon)
Ces fichiers ne sont pas dans .gitignore (qui n'exclut que *.env* implicitement via règle .env, mais ces fichiers au nom non-standard passent au travers) et sont présents depuis Initial commit. Ce sont les clés qui chiffrent les données personnelles (PII) et les IP des utilisateurs — elles doivent être régénérées immédiatement et purgées de tout l'historique git (git filter-repo/BFG), pas juste supprimées dans un commit futur.

1.2 Incohérence architecturale JWT : cookies HttpOnly vs localStorage
Le code annonce fièrement dans lib/api.ts :

Copy// 🛡️ INVESTOR-GRADE SECURITY: Cookie-First Architecture
// No localStorage, No Authorization headers.
Mais dans le même frontend, au moins 10 fichiers (admin/shifts, admin/staff, admin/cms/pages, admin/performance-reviews, admin/activity-logs, role-selector.tsx, dashboard/layout.tsx, websocket.ts) lisent activement localStorage.getItem('access_token') et l'envoient en Authorization: Bearer. Comme rien n'écrit jamais access_token dans localStorage (confirmé par grep), toutes ces pages admin sont soit mortes/cassées en prod, soit reposent sur une architecture d'auth parallèle non documentée. C'est le signe classique de deux systèmes d'auth développés à des moments différents (probablement par des passes IA successives) sans jamais être unifiés.

1.3 Fichiers *_snippet.py en prod
apps/users/cookie_views_snippet.py — un fichier nommé "snippet" avec AllowAny sur un endpoint de refresh token, jamais importé nulle part (grep ne trouve aucune référence à cookie_views_snippet dans urls.py ou ailleurs). Code mort orphelin ou copié-collé jamais nettoyé — dans les deux cas, du bruit dangereux à garder dans un module users.

1.4 Tests de sécurité qui échouent, mais jamais corrigés
backend/security_test_report_v8.txt (le dernier en date d'une série de 8 !) :

CopyFAILED tests/security/test_csrf.py::test_csrf_token_required_for_post
assert 200 == 403
Le test dit littéralement que la protection CSRF ne bloque pas les POST sans token. Le fichier a été généré, puis renommé _v2 à _v8 sans jamais corriger le problème sous-jacent — juste ignoré.

1.5 AllowAny disséminé sans revue centralisée
12 fichiers utilisent AllowAny directement dans le code des vues plutôt que via une politique de permissions centralisée et auditable. Sur un projet qui gère paiements (BaridiMob, cartes bancaires), séquestre (EscrowHold), KYC et litiges, ce pattern décentralisé est un facteur de risque élevé pour l'oubli.

1.6 Webhooks paiement : la vérification de signature est correcte, mais fragile
Le code de webhooks.py (BaridiMob/BankCard) montre une bonne pratique réelle (idempotence par event_id, select_for_update, transaction atomique, vérification de montant) — c'est un des points forts du projet. Mais :

BaridiMobService.SECRET_KEY lève une RuntimeError au chargement du module si absent — cela veut dire qu'importer apps.payments.services n'importe où (ex. dans un test qui ne configure pas cette variable) fait planter tout le processus Django, pas juste l'appel de fonction. C'est une pratique fragile pour du code de "vérification cryptographique au démarrage".

🟡 7. Autres observations notables

✅ Points positifs réels (à ne pas jeter avec le bain)
Le middleware RLS Postgres (rls_middleware.py) et son usage de set_config pour isoler le contexte utilisateur est une bonne pratique, rare à voir correctement implémentée.
La logique des webhooks de paiement (idempotence, locking, vérification de montant) est du code sérieux, pas du remplissage.
Le durcissement des settings Django (SECURE_HSTS, CSP_*, refus de démarrer en prod sans clés, refus de SQLite en prod) est mieux pensé que la moyenne.
Le séparateur JWT_SIGNING_KEY != SECRET_KEY avec un RuntimeError explicite en cas de collision est une excellente pratique défensive, rarement vue.
